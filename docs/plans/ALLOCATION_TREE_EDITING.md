# Plan: Allocation Tree Editing — Three-Case UX Enhancement (Revised)

## Context

Implements differentiated UX for three allocation tree topologies on the Edit Project allocations tab.
This is a **revised** version of `docs/plans/ALLOCATION_TREE_EDITING.md` that corrects three flaws
identified in feedback review before any code is written.

---

## Three Flaws Fixed vs. Original Plan

### Flaw 1 — `children_sum` causes false positives for shared-pool trees (Case 2a)

**Root cause:** `allocation.children` contains **shared-pool (inherited) allocations** — each child
gets the **same amount** as the parent (confirmed in `allocations.py:90` docstring and
`test_allocation_tree.py:222–235`). Summing `allocation.children.amount` always yields
`n × parent.amount`, which always exceeds `parent.amount` when n > 1 → perpetual false warning.

For Case 2b (partitioned children like CESM0002/Derecho), the sub-projects have **standalone**
allocations (`parent_allocation_id IS NULL`) that do NOT appear in `allocation.children` at all.
The original `get_children_allocation_sum()` would return 0 for the very case it was meant to catch.

**Fix:** Replace `get_children_allocation_sum(allocation)` with
`get_partitioned_descendant_sum(session, allocation)` that queries descendant projects' accounts
for non-deleted, non-inherited (`parent_allocation_id IS NULL`) allocations on the same resource.
The edit template shows the partitioned sum section **only** when this sum > 0 (i.e., only when
standalone descendant allocations actually exist). The Case 2a shared-pool cascade notice is kept
separate and does NOT display any sum.

### Flaw 2 — Detach does not decouple usage roll-up

**Root cause:** Usage roll-up uses the **project tree MPPT** (`tree_root/tree_left/tree_right`),
not the allocation `parent_allocation_id` tree (`rolling_usage.py:208`, `fstree_access.py:10–11`).
Detaching an allocation has zero effect on whether the child project's usage rolls up to the parent.

**Fix:** Add a prominent warning paragraph inside the "Break inheritance" collapse section explaining
that usage generated by the sub-project still rolls up to the parent project regardless of detachment,
and that the parent going OVERSPENT will suspend the sub-project.

### Flaw 3 — UX loop: detached children counted as "unlinked"

**Root cause:** `unlinked_descendants_count` filters by `existing_child_project_ids` built from
`allocation.children`. A detached child is no longer in `allocation.children` → counted as
"unlinked" → "Apply to remaining" button appears → `propagate_allocation_to_subprojects` runs →
`skip_existing=True` skips the detached allocation → success message says "Created 0, Skipped 1"
→ UI reloads and still shows the button. Infinite confusing loop.

**Fix:** Determine unlinked status by checking whether the descendant project has **any** non-deleted
allocation for the resource (`Account.get_by_project_and_resource` + non-deleted allocation check),
not by membership in `allocation.children`. A detached child still has an allocation → not unlinked.
Same fix applies in `htmx_propagate_to_remaining` to keep the two in sync.

### `get_partitioned_descendant_sum(session, allocation) -> float` (**replaces** `get_children_allocation_sum`)

**Why different from original:** `allocation.children` contains shared-pool copies (same amount as
parent — always wrong to sum). Partitioned allocations are standalone (`parent_allocation_id IS NULL`)
and do NOT appear in `allocation.children`. Must query descendant projects' accounts directly.

```python
def get_partitioned_descendant_sum(session, allocation) -> float:
    """
    Sum the amounts of non-deleted, non-inherited (parent_allocation_id IS NULL)
    allocations on descendant projects for the same resource as `allocation`.

    This is the correct "Case 2b" check: descendant projects that were given their
    own standalone allocation (not linked via parent_allocation_id).

    Returns 0.0 if allocation has no account, no descendants, or none have
    standalone allocations for this resource.

    NOTE: Do NOT use allocation.children for this — those are shared-pool copies
    with the same amount as the parent and would always give a false overage.
    """
    from sam.accounting.accounts import Account

    if not allocation.account:
        return 0.0

    resource_id = allocation.account.resource_id
    project = allocation.account.project
    if not project or not project.has_children:
        return 0.0

    total = 0.0
    for desc in project.get_descendants():
        if not desc.active:
            continue
        account = Account.get_by_project_and_resource(session, desc.project_id, resource_id)
        if not account:
            continue
        for a in account.allocations:
            if not a.deleted and a.parent_allocation_id is None:
                total += a.amount
    return total
```

(`propagate_allocation_to_subprojects`, `detach_allocation`, `get_partitioned_descendant_sum`).

### 4g. New route: `htmx_propagate_to_remaining` (POST) — revised for Flaw 3

```python
@bp.route('/htmx/propagate-allocation-to-remaining/<int:alloc_id>', methods=['POST'])
@login_required
@require_permission(Permission.EDIT_PROJECTS)
def htmx_propagate_to_remaining(alloc_id):
    allocation = db.session.get(Allocation, alloc_id)
    if not allocation or allocation.is_inheriting:
        return '<div class="alert alert-danger">Invalid allocation.</div>', 400
    project = allocation.account.project
    resource_id = allocation.account.resource_id

    # Flaw 3 fix: exclude descendants that already have ANY allocation for this resource
    from sam.accounting.accounts import Account
    def _has_any_alloc(proj_id):
        acct = Account.get_by_project_and_resource(db.session, proj_id, resource_id)
        return acct is not None and any(not a.deleted for a in acct.allocations)

    descendants = [
        d for d in project.get_descendants()
        if d.active and not _has_any_alloc(d.project_id)
    ]
    try:
        with management_transaction(db.session):
            created, skipped = propagate_allocation_to_subprojects(
                db.session, allocation, descendants,
                user_id=current_user.user_id, skip_existing=True,
            )
    except Exception as e:
        return f'<div class="alert alert-danger">Error: {e}</div>', 400
    return htmx_success_message(
        {'closeActiveModal': {}, 'reloadAllocationTree': project.projcode},
        f'Created {len(created)} child allocation(s). {len(skipped)} skipped (already existed).',
    )
```

## Edge Cases

| Case | Mitigation |
|---|---|
| Child already exists when "Apply to subprojects" | `skip_existing=True`; skipped still inserted into `alloc_map` |
| `break_inheritance=True` but not inheriting | Falls through to normal `update_allocation()` — no-op |
| **`partitioned_sum` for shared-pool (Case 2a)** | Returns 0.0 — all descendants' allocations have `parent_allocation_id IS NOT NULL`, so the `IS NULL` filter excludes them. No false warning shown. |
| **`partitioned_sum` for Case 2b** | Correctly sums standalone descendant allocations (not in `allocation.children`) |
| **Detached child in `unlinked_descendants_count`** | Detached child still has an allocation on the resource → `_has_any_alloc()` returns True → not counted as unlinked |
| Inactive intermediate project | Excluded by `if d.active`; `alloc_map.get(child.parent_id)` returns None → `parent_allocation_id=None` (acceptable) |
| Deep project tree (grandchildren) | `get_descendants()` DFS pre-order + `alloc_map` — unchanged |
| Propagation partial failure | Inside caller's `management_transaction()` — all-or-nothing |
| Detach+edit SQLAlchemy identity map | `session.flush()` in `detach_allocation()` updates identity map before `update_allocation()` |
| `_reload_edit_form()` missing new context vars | Recompute `partitioned_sum` and `parent_info`; pass `unlinked_descendants_count=0` on error re-renders |
| Case 2b amount validation | Advisory warning only (non-blocking) — intentional |
| Universal usage roll-up after detach | **Prominently warned in UI** — detach does not change MPPT roll-up behavior |
