Based on a thorough examination of the @src/webapp/** directories, you have built a very robust and feature-rich application. The transition to HTMX is well-implemented, and the API is logically structured. 

  However, because HTMX acts as a bridge between frontend UI and backend logic, it has introduced a "parallel track" to your REST API. This has led to several areas of duplication and maintenance bottlenecks that should be
  addressed before the codebase grows further.

  Here are the key patterns of duplication, bottlenecks, and recommendations for refactoring:

  1. Manual Form Validation vs. Schema Validation
  The Bottleneck:
  Your API routes (like api/v1/charges.py) elegantly use Marshmallow for schema validation. However, almost every HTMX endpoint (in facilities_routes.py, orgs_routes.py, projects_routes.py, resources_routes.py) manually
  extracts, casts, and validates form data.

  Example from dashboards/admin/resources_routes.py:

   1 commission_date = None
   2 if commission_str:
   3     try:
   4         commission_date = datetime.strptime(commission_str, '%Y-%m-%d')
   5     except ValueError:
   6         errors.append('Invalid commission date format.')
   7 else:
   8     errors.append('Commission date is required.')
  This pattern is repeated dozens of times across the application, leading to massive view functions that are prone to typos, missed edge cases (like handling empty strings "" vs None), and duplicated logic.

  The Fix:
  Standardize on WTForms or Marshmallow for your HTMX routes. You can define a form/schema once and use form.validate_on_submit() or schema.load(request.form). If validation fails, pass the form/schema errors directly to
  the template.

  2. Massive Template Boilerplate in HTMX Modals
  The Bottleneck:
  You have dozens of individual template files for HTMX forms (create_facility_form_htmx.html, edit_aoi_form_htmx.html, create_machine_form_htmx.html, etc.). 
  Almost every one of these files contains the exact same boilerplate:

    1 <form hx-post="..." hx-target="..." hx-swap="..." hx-disabled-elt="...">
    2     <div class="modal-body">
    3         <!-- Fields -->
    4         {% if errors %}
    5         <div class="alert alert-danger mt-2 mb-0">...</div>
    6         {% endif %}
    7     </div>
    8     <div class="modal-footer">
    9         <button type="button" class="btn btn-secondary" data-bs-dismiss="modal">Cancel</button>
   10         <button type="submit" class="btn btn-success"><i class="fas fa-plus"></i> Create/Save</button>
   11     </div>
   12 </form>

  The Fix:
  Create a reusable Jinja macro in dashboards/fragments/modal_form.html:

   1 {% macro htmx_modal_form(post_url, target, submit_text, submit_icon='save', is_edit=False) %}
   2 <form hx-post="{{ post_url }}" hx-target="{{ target }}" hx-swap="innerHTML">
   3     <div class="modal-body">
   4         {{ caller() }}
   5         <!-- Shared error rendering here -->
   6     </div>
   7     <!-- Shared footer here -->
   8 </form>
   9 {% endmacro %}
  This will cut your template sizes by 40-50% and ensure consistency across the UI.

  3. Redundant "Success" Fragments
  The Bottleneck:
  Files like facility_edit_success_htmx.html, organization_edit_success_htmx.html, resource_edit_success_htmx.html, and project_create_success_htmx.html contain nearly identical HTML:

   1 <div class="modal-body text-center py-4">
   2     <i class="fas fa-check-circle text-success fa-2x"></i>
   3     <p class="mt-2 text-success">Saved successfully.</p>
   4 </div>

  The Fix:
  Consolidate these into a single fragments/success_message_htmx.html that accepts a message parameter. The specific HTMX headers (HX-Trigger) that tell the frontend to reload cards are already being sent via Python in
  webapp.utils.htmx.htmx_success(), so the HTML templates are fully interchangeable.

  4. Duplicate Search/Autocomplete Endpoints
  The Bottleneck:
  Because you needed different FK selections and user searches, you have multiple endpoints doing the exact same backend query:
   - /htmx/search-users-for-org (in orgs_routes.py)
   - /htmx/search-users-impersonate (in admin/blueprint.py)
   - /htmx/search-users (in resources_routes.py)
   - /htmx/search-users (in user/blueprint.py)

  They all essentially call sam.queries.users.search_users_by_pattern and differ only slightly in the HTML they return (user_search_results_fk_htmx.html vs user_search_results_htmx.html).

  The Fix:
  Create a unified /htmx/search/users endpoint that accepts an optional context or template query parameter. This allows one endpoint to serve the autocomplete results for project members, impersonation, and FK pickers
  based on context.

  5. Inline Database Queries (Violating the Refactoring Plan)
  The Bottleneck:
  Your REFACTORING_PLAN.md explicitly states: "Priority 1: Centralize Data Access & Business Logic". However, the new admin HTMX routes have introduced massive inline ORM queries directly in the view layer. 

  Example from dashboards/admin/orgs_routes.py:

   1 institution_types = db.session.query(InstitutionType).options(
   2     selectinload(InstitutionType.institutions)
   3         .selectinload(Institution.users)
   4         .selectinload(UserInstitution.user)
   5         .lazyload(User.accounts),
   6     # ... more loads ...
   7 ).order_by(InstitutionType.type).all()

  The Fix:
  Move these complex query().options() chains into sam.queries.organizations, sam.queries.resources, etc. Your views should only be calling get_organization_tree(db.session) or get_resource_hierarchy(db.session). This
  keeps your views thin and makes testing the queries significantly easier.

  6. Duplication of API and HTMX Route Logic
  The Bottleneck:
  If you look at "Edit Allocation", you have:
   1. api/v1/allocations.py -> @bp.route('/<int:allocation_id>', methods=['PUT'])
   2. dashboards/user/blueprint.py -> @bp.route('/htmx/edit-allocation/<int:allocation_id>', methods=['POST'])

  Both endpoints verify permissions, extract dates and amounts, validate them, catch exceptions, and call sam.manage.update_allocation(). 

  The Fix:
  If you consolidate input parsing (using WTForms/Marshmallow as suggested in #1), you can have a single "Service" function in the web layer that takes validated data and executes the update. Alternatively, for simpler
  entities, your HTMX frontend can actually send PUT requests directly to the API, and the API can use Content Negotiation (checking if request.headers.get('HX-Request')) to return an HTML fragment instead of JSON.

  7. Re-implementation of Date Parsing
  The Bottleneck:
  You have built excellent helpers in api/helpers.py (parse_input_start_date and parse_input_end_date) that ensure end dates hit 23:59:59 to match the SAM convention. However, many HTMX routes bypass this and call
  datetime.strptime(start_str, '%Y-%m-%d') directly, which risks breaking that convention.

  ---

  Suggested Next Steps (Prioritized)

   1. Unify HTMX Modals (High Impact, Low Effort): Create a Jinja macro for your HTMX forms and a single generic success fragment. This will immediately delete hundreds of lines of duplicate HTML.
   2. Move Queries out of Admin HTMX routes (High Impact, Med Effort): Migrate the heavy selectinload queries from orgs_routes.py, resources_routes.py, and facilities_routes.py into the sam.queries module.
   3. Standardize Form Validation (High Impact, Med Effort): Implement a base form handler (like WTForms or Marshmallow) for the HTMX POST endpoints to remove the massive if not ...: errors.append(...) blocks.
   4. Consolidate Search Endpoints (Low Effort): Merge the 4 different htmx_search_users endpoints into a single configurable endpoint.
