# Alembic management for `system_status` — PR-A (Phases 1 + 2)

## Context

The `system_status` ORM at `src/system_status/` defines its own schema (unlike `sam`, which mirrors a legacy production schema we don't own). Today the schema is bootstrapped three different ways — `scripts/setup_status_db.py` for dev, `db.create_all(bind_key='system_status')` in `tests/conftest.py`, and presumed manual DDL in prod — with no migration tooling, no version tracking, and no path to evolve the schema safely across MySQL (local) and Postgres (`csg-postgres.k8s.ucar.edu`).

This PR introduces Alembic to manage `system_status` and uses it to perform a long-pending normalization: lifting `system_name`, `queue_name`, `filesystem_name`, and `node_name` out of denormalized text columns into FK references against four new lookup tables. The Alembic layout is chosen so a future `migrations/sam/` env can be added without restructuring once the legacy SAM database is retired.

A follow-up **PR-B** will add a `queue_history` table (with logical user/project linkage to `sam`) and update the collectors to populate it. PR-B is sketched at the end of this plan but not implemented in PR-A.

## Decisions (confirmed)

| Decision | Choice |
|---|---|
| Migration layout | Per-bind directory: `migrations/system_status/` (room for future `migrations/sam/`) |
| Phase 2 scope | All four lookup tables: `systems`, `queues`, `filesystems`, `login_nodes` |
| Phase 2 cut-over | Single migration: add FKs → backfill → drop old text cols → rebuild UKs |
| PR scope | Split: PR-A = phases 1+2 (this plan); PR-B = phases 3+4 (history + collectors) |
| Test strategy | Hybrid: `db.create_all` stays for per-test fixture; add one drift test that runs migrations against an empty SQLite and compares to `StatusBase.metadata` |
| Cross-DB FK in PR-B | Logical (string) only — `queue_history.upid` / `.projcode` are indexed strings, not enforced FKs |

    bind.execute(sa.text("UPDATE resource_reservations SET system_id = (SELECT system_id FROM systems WHERE name = resource_reservations.system_name)"))

    for tbl, col in [
        ('queue_status','system_id'), ('queue_status','queue_id'),
        ('filesystem_status','system_id'), ('filesystem_status','filesystem_id'),
        ('login_node_status','system_id'), ('login_node_status','login_node_def_id'),
        ('system_outages','system_id'), ('resource_reservations','system_id'),

        b.drop_index('ix_queue_status_system_name')
        b.drop_column('queue_name')
        b.drop_column('system_name')
        b.create_unique_constraint('uq_queue_status_timestamp_queue_id', ['timestamp', 'queue_id'])
    with op.batch_alter_table('filesystem_status') as b:

        b.drop_constraint('uq_filesystem_status_timestamp_filesystem_name', type_='unique')
        b.drop_index('ix_filesystem_status_filesystem_name')
        b.drop_index('ix_filesystem_status_system_name')
        b.drop_column('filesystem_name')
        b.drop_column('system_name')
        b.create_unique_constraint('uq_filesystem_status_timestamp_filesystem_id', ['timestamp', 'filesystem_id'])
    with op.batch_alter_table('login_node_status') as b:

        b.alter_column('login_node_def_id', nullable=False)

        b.drop_index('ix_login_node_status_system_name')

**Cross-dialect notes baked into the migration:**
- All ALTERs go through `batch_alter_table` (mandatory for SQLite, no-op-equivalent on MySQL/Postgres).
- All UPDATEs use correlated subqueries, never `UPDATE ... FROM` (Postgres-only) or `UPDATE ... JOIN` (MySQL-only).
- `Enum(..., native_enum=False)` avoids the Postgres `CREATE TYPE` divergence.
- Constraint names match the naming convention added in Commit 1.

**MySQL implicit-DDL-commit hazard:** documented in the migration's docstring and in PROD_BOOTSTRAP.md. Backup-before-migrate is mandatory. Backfill is small (correlated UPDATEs against a few thousand rows) and completes in seconds at current volumes.

## Risks (PR-A)

1. **MySQL implicit-commit-on-DDL.** A failure mid-migration on MySQL leaves the schema torn (no rollback). Mitigated by mandatory pre-migration backup, short backfill (seconds at current volumes), and the documented runbook. Postgres handles DDL+DML in one transaction.
2. **SQLite batch_alter_table fragility.** Phase 2 drops columns and rebuilds UKs; SQLite recreates the table for any non-trivial ALTER. Mitigation: dedicated round-trip test (`upgrade head → downgrade base → upgrade head`) on SQLite as the canonical batch-mode smoke. Verify no inbound FKs reference `queue_status.queue_status_id` etc. before authoring 0002 (none currently expected).
3. **Collector cut-over window.** ~5 minutes of dropped status snapshots during the migration. Mitigated by the documented runbook; collectors are idempotent on failure, next tick retries.
4. **Drift between Alembic envs in repo (future-facing).** Once `migrations/sam/` lands, devs may forget which env they're upgrading. Mitigation: explicit `make migrate-status-*` targets land in PR-A; `make migrate-sam-*` will follow. CI pre-commit guard (future PR) can fail if a model file changes without a corresponding migration in the same bind dir.

    __bind_key__ = 'system_status'
    __tablename__ = 'queue_history'

    queue_history_id   = Column(BigInteger, primary_key=True, autoincrement=True)
    timestamp          = Column(DateTime, nullable=False, index=True)
    system_id          = Column(Integer, ForeignKey('systems.system_id'), nullable=False, index=True)
    queue_id           = Column(Integer, ForeignKey('queues.queue_id'),  nullable=False, index=True)

    pbs_job_id         = Column(String(64), nullable=False, index=True)
    job_state          = Column(String(2), nullable=False)

    # Logical FKs (cross-DB to sam — strings, not enforced)
    upid               = Column(Integer, nullable=True, index=True,
                                comment='Logical FK to sam.users.upid; not enforced (cross-database).')
    username           = Column(String(32), nullable=False, index=True)
